API index

以下為I1系統的API，API依模組分類後以表格式方式呈現，表格的內容如下

  • API: API 的 url

  • Name: 中文名稱

  • Table: 對應的主表格名稱(Database Table)

每個API提供的參數跟操作(GET, POST, PATCH, DELETE)各有不同，點連結進入後可進入各模組的API詳情頁， 索引表只提供 API 對應的主要table名稱，table下的column需進入API詳情頁查看，SD規格書撰寫時是以Database的table name及column name為基準， API的使用者可透過映射表找到對應API

JSON 格式說明

如果是多筆的資料，回應的json會是陣列式結構，可能會有分頁資訊

{
  "content" : [ {
  	"uuid": "001",
  	"name": "item1"
  }, {
  	"uuid": "001",
  	"name": "item2"
  }, {
  	"uuid": "001",
  	"name": "item3"
  } ],
  "totalElements" : 3,
  "last" : true,
  "totalPages" : 1,
  "size" : 0,
  "number" : 0,
  "first" : true,
  "sort" : null,
  "numberOfElements" : 3
}

有像上述的 totalElements, numberOfElements, totalPages…​ 屬性的即為分頁後的結果, 屬性定義如下:

  • totalElements : 總筆數

  • last : 是否為最後一頁

  • totalPages : 總頁數

  • size : 每頁筆數

  • number : 頁碼 (從0算起)

  • first : 是否為第一頁

  • sort : 排序(目前無效)

  • numberOfElements : 本頁筆數

如果取回的api包含有換頁訊息的json，表示應api支援換頁控制， 可以透過傳入 size 來決定取回的筆數，預設為100筆，也可以透過page來決定取回第幾頁的資料

Field Required Type default Format Description

page

*

number

0

分頁頁碼(0-base)

size

*

number

100

每頁筆數

ex:

# 取回前100筆資料
/datas

# 取回前30筆資料
/datas?size=30

# 取回第5頁的資料, 預設筆數為100筆，所以是取回第 500~600 筆的資料
/datas?page=5

# 取回第二頁的資料，每頁10筆也就是取回查詢結果中的第 20~30 筆
/datas?size=10&page=2

日期时间格式

  • json response的一律为 timestamp (unit time), 以北京时间为准(GMT+8),不需要再做TimeZone 的调整

  • request 的时间每个 API 上会有格式规范 ex: (yyyy-MM-dd HH:mm:ss, yyyy-MM-dd hh:mm,…​), 以 API 给的格式传入即可

后台的 OS, Web Server, Application Server, Database等都是设定为北京时间,前台 client 在调用 API 时, 所在地若是在北京时区则不需要再做调整

排序(Sorting)

注意: 目前只有少數API支持排序，而且文檔上沒特別說明

  • 排序的語法是在 url parameter加上 sort=屬性1,屬性2,屬性3

  • 屬性前面可加上 + 表示昇冪(ASC), - 表示降冪(DESC),不加 + 或 - 時是 default 為 +

ex:

# 先以``firstName``做 ASC 排序，再以``lastName``做 DESC 排序，最後再以``age``做ASC 排序
http://www.api.com/persons?sort=firstName,-lastName,+age

對多筆的內容進行過濾

像一些query api,返回的都是list of objects,大概像这样

{
  "content" : [ {
     "uuid": ...
     "no": ...
   },
  {
     "uuid": ...
     "no": ...
   }
],
  "totalElements" : 2,
  "totalPages" : 1,
  "last" : true,
  "numberOfElements" : 2,
  "first" : true,
  "sort" : null,
  "size" : 100,
  "number" : 0
}

如果想要返回所有objects的uuid,要将content包含在includeFields,不然content就会被filter给过滤掉,里面的objects也就会被忽略

curl 'http://admin:123@localhost:8080/items?includeFields=content,uuid'

返回结果:
{

"uuid" : "F3F3AFC2-DE00-401F-842F-A8269FDC891B"
}, {

錯誤訊息格式

錯誤訊息格式可分為三類:

  1. 通用錯誤訊息

  2. 驗証失敗訊息

  3. 認證失敗訊息

特別注意: 所有的錯誤訊息的message都是針對開發人員，不應該被直接拿來顯示在終端設備上，使用api時，應該以code為key，然後轉譯成對使用者比較友善的錯誤訊息提示內容

通用錯誤訊息

通用錯誤訊息包含了以下屬性

  • status : http status code

  • code : 錯誤訊息代碼

  • message : 給"開發人員"的訊息

  • content : 額外的輔助資訊

status為HTTP 狀態碼, 每個狀態碼有其代表的意義， api常用到的status如下:

  • 200 (OK) 成功

  • 201 (CREATED) 成功建立新資源

  • 204 (No Content) 如果使用這個status code，response裡沒有任何內容

  • 400 (Bad Request) 通常發生在驗証失敗

  • 404 (Not Found) 資源不存在

  • 500 ( Internal Server Error) 伺服器錯誤

以下為幾個常見幾個通用訊息的code，每個api會視情況有自已的code，如果api有自已的code，api會特別註明

code : ResourceNotFound 找不到對應的資源

{
  "status" : 404,
  "message" : "Resource Not Found",
  "code" : "ResourceNotFound",
  "content" : "resource Item@001 not found"
}

code : IllegalApiUsage api的用法錯誤

{
  "status" : 400,
  "message" : "Illegal Api Usage",
  "code" : "IllegalApiUsage",
  "content" : null
}

code : Unknown 無法辨視的錯誤

{
  "status" : 500,
  "message" : "Unknown internal server error",
  "code" : "Unknown",
  "content" : ""
}

驗証失敗訊息

驗証失敗訊息的code是 BindingException ，結構與通用訊息一致，錯誤的細節會在 content 裡，驗証失敗依是否與欄位相關可分為 fieldErrors 跟 globalErrors

  • globalErrors: 與欄位較不直接相關的邏輯驗証失敗, 路徑位於 content.globalErrors ，用以顯示商業邏輯上的錯誤

  • fieldErrors : 與欄位直接相關的欄位驗証失敗(常見於表單驗証失敗), 路徑位於 content.fieldErrors ，裡面會有欄位名稱( content.fieldErrors.field ), 用於指明是那個欄位的錯誤

globalErrors :

主要的code為 BindingException , status為 400 ，但真正有用的error code是在’content.globalErrors.code'

{
  "status" : 400,
  "message" : "There is a validation error, check content for detail",
  "code" : "BindingException",
  "content" : {
    "globalErrors" : [ {
      "code" : "custom error code",
      "message" : null
    } ],
    "fieldErrors" : [ ]
  }
}

fieldErrors:

主要的code為 BindingException , status為 400 ，但真正有用的error code是在’content.fieldErrors.code'

  • content.fieldErrors.field : 欄位名稱

  • content.fieldErrors.code : 錯誤代碼

  • content.fieldErrors.message : 給"開發人員"的訊息

  • content.fieldErrors.rejected : 被拒絕的值

{
  "status" : 400,
  "message" : "There are 6 validation errors, check content for detail",
  "code" : "BindingException",
  "content" : {
    "globalErrors" : [ ],
    "fieldErrors" : [ {
      "field" : "no",
      "code" : "NotBlank",
      "rejected" : null,
      "message" : "may not be empty"
    }, {
      "field" : "name",
      "code" : "NotBlank",
      "rejected" : null,
      "message" : "may not be empty"
    }, {
      "field" : "status",
      "code" : "NotIn",
      "rejected" : "9",
      "message" : "must one of {1,2,3}"
    } ]
  }
}

此例中，驗証時，三個欄位發生了驗証失敗

  1. no 欄位，沒有通過 NotBlank (必填)的驗証，實際的內容(rejected)是 null

  2. name 欄位，沒有通過 NotBlank (必填)的驗証，實際的內容(rejected)是 null

  3. status 欄位，沒有通過NotIn(多選一)的驗証，實際的內容(rejected)是 9，但應為1,2或3其中一個值

如果API有多個 response，通常會有特定的error, 前端要特別針對這些 error code 做處理

認證失敗訊息

如果沒有經過認證就直接存取資源，會收到下列格式的訊息，status code可能為401或403

{
  "timestamp" : 1441291433958,

檔案上傳

上傳的動作，分成兩動:

  1. 上傳檔案，並cache在server，server會回應一個檔案的 uuid (a.k.a file uuid) 給前端，此時已可以透過取到得資訊顯示暫存在cache區的圖片，這個動作可用來做preview圖片的動作

  2. 利用步驟1取的 file uuid ，跟其他資源的 uuid 做關聯，關聯成功後，圖片會從上傳的 cache 區，移至資源指定的位置(目前是copy，之後可以會改成move，所以一旦關聯成功後，就不要再使用上傳暫存區的圖檔了)

權限設計

角色-帳號管理

角色階層

這些都是跟角色直接相關的結構，而且有階層性，所以定義為角色階層

  1. 職能(職能部門) (Fucntion)

  2. 角色 (Role)

  3. 帳號 (User)

資源操作權限

資源操作權限是指對下列的項目是否擁有權限，ex:

  • 擁有商品管理模块的權限，才能看到商品管理模块

  • 擁有预订单 - 新增的權限，才能對預訂單進行新增

資源操作權限可分為四大類，有些大類會再進細分:

  1. 資源庫 (Res)

    • 模块：商品管理模块、预订单模块

    • 功能模块：目录数据管理、目录模板管理

    • 页签：清单、表单

  2. 菜單 (Menu)

    • 新增

    • 編輯

    • 刪除

    • 審核

    • 發佈

    • 生效/失效

    • …​

  3. 元件 (CTP)

    • 畫面上的控件

  4. 數據 (Data)

    • 管控可以看到的數據量跟內容刪除

權限分配是指將資源(RES, MENU, CTP, DATA) 分配給那個角色階層(CTRL_RESOURCE_UUID), 角色階層是指來自(職能/角色/賬號)其中一個表的某一筆record，至於來自那個表，是由權限分類(TYPE)決定 權限的分配，可透過角色階層(職能/角色/賬號)進行分配，不同種類的資源權限會記錄在上述中不同權限分配表中(AAM_DETAIL_XXX)的權限分類 TYPE 跟(職能/角色/賬號) CTRL_RESOURCE_UUID

關聯方式

從概觀的角度來說,權限分配的動作就是把資源操作權限分配給角色階層

  • 角色階層 : 由 職能,角色,帳號三者組成

  • 資源操作權限 : 包含有對資源庫(模块,功能模块页签), 對元件(畫面上的控件), 對菜單(新增,編輯,刪除,審核…​)的操作權限

  • 權限分配: 將資源操作權限分配給哪些職能/角色/帳號

  • Function, Role 透過 FunctionRole 達到多對多的對應, 一個 Function 可以有多個 Role, 一個 Role 可以隸屬於多個 Function

权限管理详细设计

数据元件过滤 （DataPermit/DataBan）

  1. 客户端维护有一个系统资源和UUID，name的Map，任何需要过滤的请求都需要带上SysResUUID（Single），SysCptUUID（multiple）以及SysCptNames（multiple）， Map定义在客户端的util.js文件。 如果有新增的功能模块，需要把该功能模块的资源UUID定义在客户端。如果没有过滤UUID， 系统默认不进行数据的过滤

  2. 服务器测实现一个FilterQuery类，该类自动获取到客户端发送的过滤数据，包括SysResUUID， SysCptUUID和SysCptNames，供后续查询使用。由于过滤参数是存在一个Query类里面， 所有，所有的数据过滤的查询需要使用Query方式， 不能直接调用Service和Dao

  3. 在Controller中，需要把查询参数到XXXQuery中。在AbstractController中定义了一个setFilter方法， 该方法将根据过滤请求数据来获取真正需要用到的查询数据， 比如我们要过滤商品品牌， setFilter方法会根据SysResUUID, SysCptUUID获取到数据库中针对该资源配置的允许和禁止的商品品牌， 并存放在XXXQuery中，XXXQuery在build query的时候， 会根据Query对象的获取的数值来构建查询条件。

Q：如何添加更多的过滤项？ A：在AbstractController中的setFilter方法中添加需要过滤的字段即可setFilterData(FilterQuery.XXX, query); （此处可以优化为从配置文件读取）

数据元件过滤 （输出元件过滤）

  1. 在Controller类中返回结果之前， 需要把没有权限显示的字段过滤掉，具体请参考AbstractController中的fieldFilter方法。

  2. 首先获取当前用户需要屏蔽掉的字段。

  3. 然后需要判断当前返回对象的Entity中是否存在需要过滤的字段， 如果不存在则直接忽略。

  4. 如果存在需要过滤的字段，需要强制把对象中的值修改为无效 （String: "NA", BigDecimal: -99999999, Date: 0L）

  5. 客户端在显示数据的时候会判断该值是否为无效值， 如果无效则屏蔽整个字段。

  6. 为防止把无效值更新到数据库， 系统需要在更新记录的时候把所有的无效值改为NULL，这样系统会保证不会修改到这些没有权限查看的值。

setFilter方法设置SQL查询的输入条件，fieldFilter方法过滤没有权限显示的字段。

public Page<Item> listItems(@Valid ItemQuery query) {
        setFilter(query);
        return fieldFilter(itemService.findAll(query));
    }

update是时候， 需要忽略掉无效值。

ignoreUnAuthField(input);

Q：如何添加更多的过滤元件？ A：在AbstractController中的fieldFilter方法中添加需要过滤的字段即可; （此处可以优化为从配置文件读取）

左侧导航树过滤

导航树上的每个节点都属于一个系统资源， 因此只要判断每一个资源是否有对应的权限即可。

  1. 客户端的导航树模型的每一个节点都需要新增一个uuid选项， 该uuid需要和系统定义的SysResUuid一一对应， 后续如果有新增SysRes，需要对应维护客户端的信息。 （可以考虑导航树的信息重构为从服务器获取）

  2. 系统登录后需要查询该用户所有的资源权限。

  3. 对于存在权限的导航树节点即可显示。

菜单过滤

菜单的权限设计和导航树类似， 每个功能的每个菜单都对应一个SysMenu， 每个用户，角色以及职能都有不同的SysMenu权限， 每次打开一个功能节点时，首先需要获取该节点的菜单权限信息， 客户端再根据权限信息来决定 哪些菜单显示，哪些不显示。 每个菜单的SysMenuUuid都在客户端有配置，如果有新增功能菜单， 客户端需要修改， 否则默认没有菜单显示。

另外， 对于admin用户是不做任何数据过滤， 所有的数据， 导航树以及菜单等都全部显示。
