目錄管理

目錄模版簡介

上圖中，包含了三個目錄模版，每個目錄模版包含數個目錄模版節點，每個目錄模版節點底下，包含了數個目錄數據，同一個目錄模版內的目錄模版節點有順序性。

目錄模版一中，包含了按品牌 → 按類別 →按風格三個目錄節點，這三個目錄有順序性，按品牌為按類別的parent，按類別為按風格的parent。

每一個目錄模版中，最多可有一個節點被標示為默认APP目录，默认APP目录會成為app版畫面的頂級節點(根節點)。目錄節點下方為對應的目錄數據。

  1. 按品牌目錄節點底下的目錄數據為我愛我家、至白小屋跟多喜愛。

  2. 按類別目錄節點底下的目錄數據為床、椅子和衣櫃

  3. 按風格目錄節點底下的目錄數據為田園風格、歐洲風格

目錄數據間也有 parent - children關係，目錄數據是一個tree結構，彼此間也是用 parent uuid做關聯。

目录模板

目錄模版為 Master-Detail 結構,Master 為目錄模版, Detail 為目錄模版節點，取得目錄模版時會一併同時取出該目錄模版下的模版節點， 而異動時，目錄模版僅對目錄模版的部份做異動，如果要異動目錄模版節點，可透過目錄模版節點管理api進行。

每個目錄模版下可包含多個目錄模版節點，未包含任何目錄模版節點的模版為空模版。

目录模板单身

目錄模版節點管理

每個目錄模版下可包含多個目錄模版節點，節點間有順序性，每個節點以parentUuid與上一級的節點做串接，在同一模版內節點間的關係只能從下級透過parentUuid由下往上查，如果節點的parentUuid為null時則表示為根節點(第一個節點)。

同一個目錄模版下的目錄模版節點無法直接透過目錄模版節點的api取得，而是需透過目錄模版api取得單筆的目錄模版，目錄模版的details的屬性，即為目錄模版節點陣列。陣列的內容已照順序排列；如果只需取得單筆的目錄節點，則可透過目錄模版節點的uuid取得。

目錄模版下最多可有一個節點被標示為默认APP目录，如果默认APP目录标示被標示為true，那麼這個節點會成為在app版的目錄模版的根節點(所以app版第一級的目錄節點的parent uuid有可能不是null)。

details為一陣列結構，details裡的每一個element為一目錄模版節點。

目錄數據清單的取得比較特別，目錄數據是一個tree結構，api為了簡化使用，把這個tree結構的簡化成一階的list結構， 在查詢目錄數據時，需要先指定對應的目錄模版節點的uuid，這樣可以查到該節點底下的所有目錄數據

因為是tree結構，所以可以一層層往下查。，往下查一層的方式，是透過傳入下一階的目錄模版節點的uuid來達到。 查下一層級時，需要把templateDetailUuid換成下一個模版節點的uuid。但是這樣查出來的內容，可能會是很多組結果 ，需要再用上一層目錄數據的uuid當作parent uuid進行過濾，如果是根節點時，parent uuid傳入大寫的"ROOT"，其他的節點時， 則傳入上一層目錄數節的uuid當作parent uuid

商品類型說明

商品類型包含了 套件、包件、補件，其他件，以下舉例說明商品類型的差異

套件

套件通常是指可獨立販售的商品

以下的商品圖片為成長書桌

成長書桌這個商品本身是一個套件(虛擬套件)底下又包含了三個套件

  1. 桌子

  2. 椅子

  3. 櫃子

包件

包件是組成套件的部件，以上面的三個套件來說，它們的包件為

桌子為套件，有以下包件:

  1. 桌面

  2. 桌腳

  3. 桌面角度調整設備

椅子為套件，有以下包件:

  1. 椅背

  2. 椅墊

  3. 椅腳

  4. 椅套

  5. 滑輪 * 5

櫃子為套件，有下包件: 1. 椅墊 2. 抽遞 3. 收納櫃 3. 滑輪 * 6

補件

補件就是一些配件，通常不直接販售

像桌子可能有一些預備用的螺絲或卡筍

其他件

無法歸類在上述幾種類型的就歸在其他件
